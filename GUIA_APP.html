1. Visión general del proyecto

Sistema web de gestión de Formación en Centros de Trabajo (FCT) del IES Ciudad Escolar. Permite que tutores y el administrador gestionen alumnos, convenios con empresas y planes formativos.

Roles del sistema

RolQué puede hacer
adminGestionar tutores, validar convenios pendientes, ver y editar todos los convenios, importar convenios masivamente, ver listado global de alumnos
tutorGestionar sus alumnos, solicitar y editar convenios, rellenar y exportar planes formativos (FFE), subir documentos de seguimiento

Tecnologías usadas

  • PHP 8 — Backend y lógica de negocio
  • MySQL — Base de datos citye en XAMPP
  • PDO — Capa de abstracción para consultas SQL seguras
  • Tailwind CSS v4 (CDN) — Estilos visuales
  • PhpSpreadsheet (Composer) — Leer y escribir archivos Excel (.xlsx)
  • PhpWord (Composer) — Exportar documentos Word
  • Vanilla JavaScript — Interacciones del frontend (modales, AJAX)

Patrón arquitectónico: MVC

3. Cómo funciona la aplicación (flujo completo)

El router: index.php

Todo el tráfico pasa por index.php. Es el único archivo accesible directamente. Funciona con 4 bloques if-else:

// 1. Si viene logout → Controlador/Logout.php
// 2. Si hay sesión activa → determina el controlador por rol y llama a la acción
// 3. Si viene el botón de login → Controlador_Usuarios.php
// 4. Si no hay nada → muestra Vista/Login.php

¿Cómo llega la acción al controlador?

Los formularios y botones de la app envían un campo oculto name="accion". El router lo captura con $_REQUEST['accion'] y llama al método del mismo nombre en el controlador:

$accion = $_REQUEST['accion'] ?? "mostrarPanel";
// Si accion = "mostrarConvenios" → llama a $controlador->mostrarConvenios()

Flujo de una petición típica (tutor añade alumno)

→
POST index.php
accion=guardarAlumno
→
Controlador_Tutores
::guardarAlumno()
→
Modelo Alumnos
::insertar()
→
SQL INSERT
→
Vista recargada

4. Conexión a la base de datos — Singleton + PDO

¿Qué es el patrón Singleton?

Garantiza que solo existe una instancia de la conexión durante toda la petición. Si varios modelos necesitan la BD, todos reutilizan el mismo objeto PDO en vez de abrir nuevas conexiones.

class Conexion {
    private static $instancia = null;

    public static function getConexion() {
        if (self::$instancia === null) {        // Solo conecta si no hay conexión
            self::$instancia = new PDO(
                "mysql:host=localhost;dbname=citye;charset=utf8",
                "root", "",
                [PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION]
            );
        }
        return self::$instancia;
    }
}

Datos de conexión

  • Host: localhost
  • Base de datos: citye
  • Usuario: root
  • Contraseña: vacía (entorno local XAMPP)
  • Charset: utf8

¿Qué es PDO y por qué se usa?

PDO (PHP Data Objects) es la capa de abstracción estándar de PHP para bases de datos. Ventajas clave:

  • Previene SQL Injection mediante consultas preparadas con prepare() y execute()
  • Independiente del motor de BD (podría cambiar a PostgreSQL sin tocar el código)
  • Lanza excepciones en vez de fallar silenciosamente (ERRMODE_EXCEPTION)

Ejemplo de consulta segura con PDO

$sql  = "SELECT * FROM tutores WHERE dni = :dni";
$stmt = $conn->prepare($sql);           // prepara la consulta
$stmt->execute(['dni' => $dni]);        // inyecta el valor de forma segura
$result = $stmt->fetch(PDO::FETCH_ASSOC); // devuelve array asociativo

5. Sistema de seguridad y control de accesos

¿Cómo funciona la protección de vistas?

Cada archivo PHP de vista o controlador incluye al principio:

require_once $_SERVER['DOCUMENT_ROOT'] . '/PROYECTO/Seguridad/Control_Accesos.php';
validarAcceso('admin'); // o 'tutor' o 'cualquiera'

¿Qué comprueba?

  1. Que $_SESSION['usuario'] exista → si no, error de sesión no iniciada
  2. Que $_SESSION['rol'] coincida con el rol requerido → si no, error de acceso restringido
⚠️ Importante: Si alguien conoce la URL de una vista de admin e intenta acceder sin sesión o siendo tutor, la función validarAcceso() llama a die() y muestra una pantalla de error — nunca se ejecuta el código de la vista.

El login

  1. El usuario envía DNI + contraseña
  2. Controlador_Usuarios.php busca el usuario en la BD por DNI
  3. Valida la contraseña con password_verify($input, $hash) (las contraseñas están hasheadas con bcrypt)
  4. Si es válido, crea la sesión: $_SESSION['usuario'], $_SESSION['rol'], $_SESSION['id_ciclo'], etc.
  5. Redirige a index.php que ya detecta la sesión

6. Importar alumnos desde Excel

¿Cómo funciona?

El tutor descarga la plantilla (Recursos/Importar/plantilla_listadoAlumnos.xlsx), la rellena y la sube desde la pestaña de Alumnos.

Formato esperado de la plantilla

Columna AColumna BColumna C
NombreApellido(s)Dirección de correo
JUANGARCIA LOPEZjuan@email.com
ℹ️ Nota: Los apellidos van en una sola columna separados por espacio. El sistema divide automáticamente en apellido1 y apellido2 con explode(' ', $apellidos, 2).

Flujo técnico del archivo Importar_Alumnos.php

  1. Validación inicial: Comprueba que llegó el archivo y que es .xlsx o .xls
  2. Lectura con PhpSpreadsheet: IOFactory::load($tmp) carga el Excel; toArray() convierte la hoja a un array PHP
  3. Saltar la cabecera: if ($idx === 0) continue; — la fila 0 son los títulos de columna
  4. Por cada fila válida:
    • Inserta en tabla alumnos (nombre, apellidos, correo)
    • Inserta en tabla curso_academico vinculando el alumno al ciclo del tutor en sesión
    • Usa una transacción (beginTransaction / commit / rollBack) para que ambas inserciones sean atómicas
  5. Respuesta JSON: devuelve { success, insertados, omitidos, errores }
  6. El frontend muestra el resultado en el modal sin recargar la página (fetch + JSON)
🔄 ¿Por qué transacciones? Si el INSERT en alumnos funciona pero el de curso_academico falla, el rollBack() deshace el primero. Así no quedan alumnos "huérfanos" sin ciclo asignado.

7. Exportar el Plan Formativo a Excel (plantilla FFE)

¿Qué es el FFE?

El Fichero de Formación en Empresa (FFE) es el documento oficial que el tutor debe entregar con los datos del alumno, la empresa y los Resultados de Aprendizaje (RAs) a impartir.

Arquitectura de la plantilla Excel

La plantilla plantilla_ffe.xlsx tiene 3 hojas:

HojaContenido¿Se modifica?
datos fijosDatos del centro, ciclo, RAs. La columna A tiene claves (ej: nombre_ciclo) y la columna B el valor.Sí — el sistema busca la clave en col A y escribe el valor en col B
datos variablesDatos del alumno, empresa, fechas, horario. La fila 1 son cabeceras y la fila 2 los valores.Sí — el sistema lee la cabecera en fila 1 y escribe en fila 2
constantesFórmulas y celdas de presentación que calculan automáticamente basándose en las otras hojas.No — se deja intacta

¿Cómo escribe los datos el sistema?

// Hoja "datos fijos": busca la clave en columna A, escribe en columna B
function setValorHoja($ws, $key, $value) {
    for ($row = 1; $row <= $ws->getHighestRow(); $row++) {
        if ($ws->getCell("A{$row}")->getValue() === $key) {
            $ws->getCell("B{$row}")->setValue($value);
            return;
        }
    }
}
// Ejemplo: setValorHoja($ws, 'nombre_ciclo', '2º DAW')

Flujo completo de exportación

  1. El tutor rellena el formulario del Plan Formativo en la app
  2. Pulsa "Exportar" → envía todos los datos via POST a Exportar_PF.php
  3. El controlador carga la plantilla con IOFactory::load($plantilla)
  4. Rellena "datos fijos": centro, ciclo, RAs de la BD (hasta 14)
  5. Rellena "datos variables": alumno, empresa, fechas, horario, excepciones
  6. Envía el archivo al navegador con cabeceras HTTP de descarga: 
    Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet
Content-Disposition: attachment; filename="datos_ffe_APELLIDONOMBRE.xlsx"
  7. PhpSpreadsheet escribe en php://output — el archivo se descarga directamente sin guardarse en el servidor
✅ Ventaja clave: Al usar una plantilla existente, las fórmulas, el formato visual, los colores y el diseño oficial del documento se conservan exactamente. El sistema solo cambia los valores de datos.

Los Resultados de Aprendizaje (RAs)

Se obtienen de la BD con una consulta que cruza las tablas resultados_aprendizaje, modulos y plan_estudios filtrando por el id_ciclo del tutor en sesión. Se rellenan hasta 14 RAs en la plantilla.

8. Importar convenios desde Excel

El administrador puede cargar un listado masivo de empresas con convenio usando Controlador/Importar_Convenios.php.

Columnas esperadas en la plantilla

Nº Convenio* | Nombre Empresa | CIF | Dirección | Municipio | CP | País | Teléfono | Fax | Mail | Nombre Rep. | DNI Rep. | Cargo

* El Nº de convenio se ignora al importar.

Proceso

  1. El admin sube el Excel desde el modal "Importar Convenios"
  2. Se envía via fetch() a index.php?accion=importarConvenios
  3. PhpSpreadsheet lee el archivo y procesa fila a fila
  4. Para cada fila válida, inserta en la tabla convenios
  5. Devuelve JSON con insertados, errores y filas omitidas
  6. El modal muestra el resultado sin recargar la página

9. Enlace externo de registro de convenios

¿Qué es?

Una URL pública que el tutor puede enviar a las empresas para que rellenen sus datos sin necesidad de tener cuenta en el sistema.

La URL

http://localhost/PROYECTO/Convenios/Registro.php?id_ciclo=3

El parámetro id_ciclo identifica a qué ciclo (y por tanto a qué tutor) pertenece la solicitud que va a llegar.

¿Cómo funciona internamente?

La clave: El archivo Registro.php detecta si el visitante tiene sesión activa o no:
  • Sin sesión (empresa externa): El formulario envía los datos via AJAX a Procesar.php. Si hay éxito, muestra un modal de confirmación y resetea el formulario.
  • Con sesión (tutor logueado): El formulario envía a ../index.php como un POST normal con accion=guardarNuevoConvenio.

Flujo cuando la empresa rellena el formulario externo

Empresa abre el enlace con ?id_ciclo=3
→
Registro.php detecta: sin sesión
→
Empresa rellena y envía
→
fetch() → Procesar.php
→
Controlador_Registro → INSERT en convenios_nuevos
→
Modal de éxito en pantalla

¿Dónde quedan las solicitudes?

En la tabla convenios_nuevos de la BD. El tutor las ve en su pestaña de Convenios con estado "Pendiente". Puede aprobarlas (el admin las valida después) o rechazarlas.

Validación de seguridad

Si se accede a Registro.php sin el parámetro id_ciclo y sin sesión, el sistema muestra un modal de error y ejecuta die() — no se puede usar el formulario sin saber a qué ciclo pertenece.

10. Modo oscuro / claro

¿Cómo funciona el sistema de temas?

  1. Al cargar la página, un script inline en el <head> consulta localStorage: 
    if (localStorage.getItem('theme') === 'dark' || coincide-con-preferencia-OS) {
    document.documentElement.classList.add('dark');
}
    Esto evita el "parpadeo blanco" al cargar en modo oscuro.
  2. El botón (luna/sol) añade o quita la clase dark del elemento <html> y guarda en localStorage.
  3. El CSS de modo oscuro (dark-mode.css) usa el selector html.dark .clase-de-tailwind con !important para sobreescribir los estilos de Tailwind.
⚠️ Por qué NO se usaron los prefijos dark: de Tailwind: La versión del CDN usada (@tailwindcss/browser@4) aplica dark: basándose en la preferencia del sistema operativo (prefers-color-scheme), no en la clase html.dark. Por eso se usa un CSS personalizado que sí responde a la clase.

12. Preguntas frecuentes de defensa

❓ ¿Por qué todo pasa por index.php?
Es el patrón Front Controller. Centraliza la seguridad (si un archivo de vista se accede directamente, la función validarAcceso() lo bloquea) y simplifica el enrutamiento. Las URLs son siempre index.php?accion=X, fáciles de mantener.
❓ ¿Cómo se evita el SQL Injection?
Usando consultas preparadas de PDO (prepare() + execute() con parámetros nombrados :nombre). Nunca se concatena directamente el input del usuario en el SQL.
❓ ¿Por qué el Singleton en la conexión?
Para no abrir múltiples conexiones a MySQL en una misma petición. En PHP, cada petición HTTP crea una nueva ejecución, pero dentro de esa petición pueden instanciarse varios modelos. El Singleton garantiza que todos comparten el mismo objeto PDO.
❓ ¿Cómo se asegura que un tutor no vea datos de otro?
Todas las consultas del tutor filtran por id_ciclo o id_tutor que vienen de la sesión ($_SESSION['id_ciclo']), no de parámetros de usuario. El tutor no puede manipular su propio ID.
❓ ¿Qué es una transacción y cuándo se usa aquí?
Una transacción agrupa varias operaciones SQL para que o se completan TODAS o no se completa ninguna. Se usa al importar alumnos: se insertan en dos tablas (alumnos y curso_academico). Si la segunda falla, el rollBack() deshace la primera, evitando datos inconsistentes.
❓ ¿Cómo funciona la exportación a Excel sin guardar en el servidor?
PhpSpreadsheet escribe en el flujo de salida PHP (php://output). Las cabeceras HTTP le dicen al navegador que el contenido es un archivo Excel para descargar (Content-Disposition: attachment). Nunca se escribe en disco.
❓ ¿Puede cualquiera acceder al formulario de registro de convenios?
Sí, es intencionado — es una URL pública. Pero sin el parámetro id_ciclo, el sistema muestra un error y bloquea el acceso. Con el parámetro, los datos se vinculan al ciclo correcto. El enlace se envía directamente a la empresa por el tutor.
❓ ¿Cómo se almacenan las contraseñas?
Con bcrypt mediante password_hash() de PHP. En la BD hay un hash, nunca la contraseña en texto plano. La verificación es con password_verify($input, $hashAlmacenado).
❓ ¿Por qué se usa Tailwind por CDN y no compilado?
Simplifica el despliegue: no hace falta Node.js ni proceso de build. En producción lo ideal sería compilarlo, pero para un entorno académico el CDN es suficiente y acelera el desarrollo.
❓ ¿Cómo se sube un documento de seguimiento?
Via fetch() con FormData enviado a index.php?controlador=Tutores&accion=seguimientoSubir. El servidor sanitiza el nombre del archivo, verifica la extensión y lo guarda en Documentacion/CICLO/ALUMNO/tipo/. La respuesta es JSON. El frontend actualiza la lista sin recargar.
❓ ¿Qué hace el .htaccess de las carpetas internas?
Impide el acceso directo a las carpetas Core/, Modelo/, Controlador/, etc. desde el navegador. Solo son accesibles mediante require_once desde PHP en el servidor, nunca por URL directa.
❓ ¿Qué tablas principales tiene la base de datos?
  • tutores — datos del tutor y su ciclo asignado
  • ciclos y cursos — ciclos formativos disponibles
  • alumnos — datos personales del alumno
  • curso_academico — vincula alumno ↔ ciclo ↔ año
  • convenios — empresas con convenio activo
  • convenios_nuevos — solicitudes pendientes de validación
  • modulos y resultados_aprendizaje — estructura académica del ciclo
  • plan_estudios — vincula módulos con ciclos
  • usuarios — credenciales de acceso (dni + hash contraseña + rol)
